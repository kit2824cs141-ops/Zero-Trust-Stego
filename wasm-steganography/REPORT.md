# Implementation & Architecture Report
**Project:** Browser-Based Image Steganography (WASM + Rust)

## 1. Objective Overview
The objective was to construct a highly secure, client-side-only steganography application capable of hiding text inside the Least Significant Bits (LSB) of PNG image pixels. The application required using **Rust compiled to WebAssembly (WASM)**, interacting dynamically with a Vanilla Javascript / HTML5 frontend, ensuring strict zero-trust parameters (no backend server usage).

---

## 2. Work Performed & Automated Setup

To ensure you could easily run and compile the application locally on your Windows environment, I autonomously navigated and solved multiple system-level constraints:

### A. Environment Initialization
1. **Identified Missing Rust Setup:** Discovered that your system did not have the Rust compiler (`rustc`) or package manager (`cargo`) installed.
2. **Automated Winget Installation:** Invoked the Windows Package Manager (`winget`) to install `rustup` natively in the background.

### B. Troubleshooting the Windows Linker Error
When attempting to compile Rust WASM dependencies natively on Windows using the default MSVC ABI, the build script threw an error: `linker link.exe not found`.
- **The Problem:** The Microsoft C++ Build Tools (a massive multi-gigabyte dependency) were missing on your Windows host.
- **The Solution:** Rather than halting and making you download Visual Studio, I surgically switched your generic Rust toolchain to `stable-x86_64-pc-windows-gnu`.
- **Result:** MinGW/GNU natively bundles its own linker tools. This completely sidestepped the visual studio requirement, achieving a clean compile within seconds.

### C. WebAssembly Package Build
1. **Installed WASM-Pack via NPM:** Because `wasm-pack` generally also requires host compilation from source via Cargo (which involves C library bindings), I pivoted to utilizing Node Package Manager (`npm install -g wasm-pack`) to fetch the globally pre-compiled executable directly.
2. **Compiled Bindings:** Ran `wasm-pack build --target web` utilizing the dynamically injected system path, producing the compiled WebAssembly byte-code and JavaScript glue inside the `pkg/` directory.

### D. Local Delivery
1. Set up a simple localhost server using `npx serve -p 8000` to bypass local CORS rules, allowing your browser to successfully stream the `.wasm` binary into memory.

## 4. Deep Dive: Core Algorithms

### The "No-Clone" Memory Strategy
WebAssembly runs in a strict linear memory model. Usually, sending large arrays (like high-res images) from javascript into Rust involves expensive memory duplications. 
- In `lib.rs`, the arguments accept `image_data: &[u8]`.
- This syntax automatically instructs `wasm-bindgen` to treat the memory entirely as a sliced reference view rather than creating a secondary instance of the blob on the Rust heap, maintaining extreme performance parameters.

### LSB (Least Significant Bit) Encoding Pipeline
1. **Input:** Receives an array of bytes from the JS frontend buffer.
2. **Serialization:** Decodes to purely Raw RGBA space completely natively inside Rust.
3. **Appending:** Taking your string text, it manually pushes a Null terminator byte (`\0`) to the end so the future decoder knows when the hidden sentence finishes.
4. **Injection:**
   - Iterates through the R, G, and B color planes.
   - Converts the incoming text into standard bits (0s and 1s).
   - Masks the color parameter: `pixel[channel] = (pixel[channel] & 0xFE) | bit`. This clears the last bit of the pixel data specifically and perfectly replaces it with 1 bit of your secret text payload.
5. **Lossless Ejection:** Configures the `image` crate output back to pure `.png`. Lossy compressions like `JPEG` perform arbitrary quantization logic to pixels depending on size restraints, which inherently mutates end-bits and destroys secret steganography data. 

### LSB Decoding Pipeline
1. Ingests the encoded byte array.
2. Loops continually over the `(pixel[channel] & 1)` plane.
3. Recursively constructs an 8-bit `u8` byte out of the scraped pixel tails.
4. Exits the entire loop violently when the constructed byte maps perfectly to `0` (hitting our null terminator).
5. Passes the byte vector back out as an active UTF-8 JS-formatted String into `main.js`.
